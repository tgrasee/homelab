Automated Homelab
Infrastructure Stack

A fully automated home server built on Proxmox - provisioned with Terraform, configured with Ansible, monitored with Grafana, and validated on every push with a CI/CD pipeline.

See the Stack GitHub Repo

Problems Solved

Real infrastructure problems this project is built to address.

Problem

Manual setup can't be reproduced

If a server dies, rebuilding it from memory takes days - and it never comes back exactly the same. Every manual step is a potential inconsistency.

Solved by

Terraform provisions VMs and Ansible configures them. The entire server rebuilds from a single command, identically, every time.

Problem

You find out something broke when users complain

Without monitoring, outages are discovered by the wrong people at the wrong time. No visibility means no warning.

Solved by

Prometheus scrapes metrics every 15 seconds. Grafana alerts fire to email the moment a node goes down or disk fills up - before anyone notices.

Problem

One failure and the data is gone

Without automated backups, a single disk failure or misconfiguration can wipe everything. Manual backup processes get skipped under pressure.

Solved by

Proxmox Backup Server runs nightly VM snapshots automatically. No manual steps, no skipped nights.

Problem

Broken configs get deployed with no safety net

Without a CI/CD pipeline, there's nothing to catch a bad Ansible playbook or invalid Terraform config before it hits the server.

Solved by

Every push to Gitea triggers Woodpecker CI - ansible-lint and terraform validate run automatically and block bad code before it can be applied.

⚙️

Zero-Touch Operations

The entire server - from OS install to running services - is defined as code. A single command provisions VMs, installs software, and deploys all services. No manual steps, no configuration drift, fully repeatable.

Automated nightly backups run without intervention. If anything changes unexpectedly, it gets flagged.

🔔

Always-On Monitoring

Grafana dashboards track CPU, memory, disk, and network across every host in real time. Prometheus collects metrics every 15 seconds. Loki aggregates logs from all services.

Alerts fire to email the moment a node goes down or disk usage crosses 85% - before it becomes a problem.

Lessons Learned

Real problems encountered during the build and how they were resolved.

Terraform

Terraform providers are not interchangeable

The commonly documented telmate/proxmox provider is incompatible with Proxmox 9. Migrating to bpg/proxmox required rewriting the VM resource schema entirely - different attribute names, different block structures. Always verify provider compatibility with your target platform version before starting.

Ansible

Cloud images don't use root

Ubuntu cloud-init images use ubuntu as the default SSH user, not root. Ansible was silently failing to connect until the inventory was updated with ansible_user: ubuntu. Additionally, SSH host key verification must be accepted manually before any automation can connect for the first time.

Ansible

Idempotency must be explicitly designed in

Re-running an Ansible playbook overwrote the production .env file, wiping live credentials. The fix was adding force: false to copy tasks so they only create the file if it doesn't already exist. Idempotency doesn't happen automatically - every task needs to be written with re-runs in mind.

OS Compatibility

Some packages are OS-specific

Proxmox Backup Server packages depend on Debian Bookworm-specific library versions and fail to install on Ubuntu 22.04 due to dependency conflicts. This required provisioning a separate Debian 12 VM for PBS rather than reusing the Ubuntu template - a constraint that isn't obvious until you hit it.

Networking

Gitea blocks webhooks to private IPs by default

After enabling the Woodpecker CI repo, pushes weren't triggering pipelines. Gitea has a security feature that blocks outbound webhook calls to private IP ranges to prevent SSRF attacks. Since Woodpecker runs on the same local network, the webhook was silently denied until GITEA__webhook__ALLOWED_HOST_LIST was configured.

Grafana

Provisioned datasource UIDs must match exactly

Grafana auto-generates a UID when a datasource is first created. Provisioned dashboards that reference ${DS_PROMETHEUS} fail silently if that variable isn't mapped to the actual UID at import time. The fix was to pin a fixed UID in the datasource provisioning YAML and update the dashboard JSON to reference it directly.

AI Tooling

AI-assisted development accelerates without replacing understanding

Claude Code was used throughout this project to accelerate troubleshooting, generate documentation, and scaffold automation. The key insight: AI tooling is most effective when you can evaluate what it produces. Every generated config was reviewed, every error was diagnosed, and every fix was understood before being applied. The result was faster iteration without sacrificing ownership of the outcome.
